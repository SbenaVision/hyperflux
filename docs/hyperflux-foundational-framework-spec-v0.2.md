## 1. Problem Statement

HyperFlux today is an engine. It externalizes decisions — business logic, labels, config, lifecycle guards — into rules that can be changed without a redeploy. This is the right idea.

The problem: an engine alone addresses ~25–30% of development work — the decision layer. The remaining 70% — components, routing, data fetching, forms, tables, modals, navigation — is still written from scratch by every developer, every time, in whatever way they choose. The result is drift, inconsistency, and a codebase that grows with every feature.

**The goal:**
- 80% reduction in development time for new applications, measured against the benchmark
- 80% reduction in changes that require a code edit, build, and deploy
- A developer building a standard CRUD app produces fewer than 500 lines of application code

We are not meeting this goal today. The codebase has ~10,000 lines of infrastructure code and ~200 externalized rules. That ratio is inverted from where it needs to be for a developer building on top of HyperFlux.

## 2. Terminology (closed, non-negotiable)

| Term | Definition |
|---|---|
| **HyperFlux Core** | The engine: resolver, rule store, lifecycle, trace, expressions. Stable. Rarely changes. |
| **HyperFlux UI** | Rule-driven primitive components: Table, Form, Modal, Nav, Toolbar, DetailPanel. Every configurable behavior reads from rules. |
| **HyperFlux Shell** | The starter template a developer clones. Structured as 90% rules, 10% wiring. |
| **HyperFlux Component** | Any component created via `createHyperFluxComponent()`. The enforced pattern for all application UI. |
| **Value-source contract** | The rule that all configurable values — in HyperFlux components and third-party components alike — must come from rules, never from literals. |
| **Base rules** | Rule JSON files compiled into the build. The versioned safety net. Never change at runtime. |
| **Runtime overrides** | Rules written via the admin and stored in a KV/database. Shadow base rules immediately, with no build or deploy. |
| **hf migrate** | AI-powered CLI tool that scans an existing app, extracts externalizable values as draft rules, and produces a review PR. |
| **Draft rule** | A rule generated by migration or the admin, marked `status: "draft"`. Requires human promotion before becoming active. |

### 3.2 Runtime Rule Storage (the "no redeploy" architecture)

This section defines what makes the core promise true. Without it, changing a rule still requires a PR, build, and deploy. That is not acceptable.

**Two-layer rule system:**

```
Layer 1 — Base rules (JSON files in repo)
  Compiled into the build artifact.
  Versioned with code. The safety net.
  Changed only via PR. Slow path.

Layer 2 — Runtime overrides (KV store / Vercel KV / Postgres)
  Written by the admin at runtime.
  Shadow base rules on every request.
  No build. No deploy. Effective immediately.
```

The `RuntimeRuleStore` in `apps/admin/hiflux/src/lib/ruleStore.ts` already implements this two-layer pattern. The `overrides` Map shadows `sourceRules`. The `overridePersistence.ts` currently writes to local disk — in production this write goes to the configured store.

**Required store adapter interface:**
```typescript
interface RuleOverrideStore {
  get(path: string): Promise<RuntimeRule | undefined>;
  set(rule: RuntimeRule): Promise<void>;
  delete(path: string): Promise<void>;
  getAll(): Promise<RuntimeRule[]>;
}
```

Implementations: `VercelKVAdapter`, `PostgresAdapter`, `LocalFileAdapter` (dev only).

**Cache invalidation:** Overrides are loaded per-request with a configurable TTL (default: 0 in dev, 5s in production). On write, the admin invalidates the cache immediately via an internal webhook. No stale reads in the write path.

**Rollback:** The override store is append-only with timestamps. Rolling back a rule means deleting its override entry — the base rule resurfaces instantly.

**Audit:** Every write to the override store is recorded in the lifecycle engine's audit log (already built). The admin surfaces this as a change history per rule.

**Phase 0 deliverable:** Implement `VercelKVAdapter` and wire it into `overridePersistence.ts`. The admin must work against a real runtime store before Phase 1 begins. This is the load-bearing foundation.

## 4. The Enforcement Mechanism

### 4.1 The Contract

Every configurable value in a HyperFlux application — regardless of which component renders it — must come from a rule. This is the **value-source contract**.

A configurable value is: any string visible to a user, any numeric threshold that could ever change, any boolean that affects behavior, any set of options presented to a user.

A configurable value is NOT: CSS class names, HTML attribute names, component keys, `aria-*` values, `data-*` values.

### 4.2 `createHyperFluxComponent()` — Closed Decision

**Factory function. Not a HOC. Not a class decorator.**

HOC: breaks ref forwarding, produces unreadable stack traces, TypeScript generics degrade at every composition layer, cannot compose more than two levels without pain. Ruled out.

Decorator: experimental TypeScript feature, unstable across compiler versions, does not work on function components without transform configuration. Ruled out.

Factory: full TypeScript inference on the rules type parameter, clean React DevTools display names, zero ref issues, trivial test mocking, component body is indistinguishable from normal React. Selected.

**Signature:**
```typescript
function createHyperFluxComponent<TRules extends RuleMap>(
  namespace: string,
  render: (props: { rules: TRules } & React.HTMLAttributes<HTMLElement>) => React.ReactElement
): React.FC<React.HTMLAttributes<HTMLElement>>;
```

**Usage:**
```tsx
// types generated by `hf generate-types` from rules/orders/submit_button.json
interface SubmitButtonRules {
  label: string;
  disabled_reason: string;
}

const SubmitOrderButton = createHyperFluxComponent<SubmitButtonRules>(
  "orders.submit_button",
  ({ rules, order }) => {
    const canSubmit = useRule("orders.policy.can_submit", {
      order_status: order.status,
      item_count: order.items.length,
    });
    return (
      <button disabled={!canSubmit} title={!canSubmit ? rules.disabled_reason : undefined}>
        {rules.label}
      </button>
    );
  }
);
```

The factory handles: rule subscription, selective re-render on rule change, fallback on missing rule, React.memo wrapping, display name registration, test mode injection.

### 4.3 Third-Party Components — Closed Decision

**You do not wrap third-party components into HyperFlux components. You enforce the value-source contract at the call site.**

A shadcn `Button` with a hardcoded label violates the contract. A shadcn `Button` whose label comes from `useRule()` satisfies it. The component type is irrelevant. The value source is what matters.

```tsx
// Violation — lint rule catches this
<Button>Submit Order</Button>

// Compliant — value comes from a rule
const label = useRule("orders.submit_button.label");
<Button>{label}</Button>

// For repeated use, createHyperFluxAdapter() wraps the pattern
const HFButton = createHyperFluxAdapter(Button, "orders.submit_button");
// internally calls useRule for all mapped props
```

`createHyperFluxAdapter(ExternalComponent, namespace)` binds an external component to a rule namespace, mapping rule values to the component's props. The lint rule allows adapter-wrapped external components and enforces that their configurable props come from the adapter, not from literals.

### 4.4 ESLint Enforcement — `@hyperflux/eslint`

Two rules. Both are hard errors in CI. Both are warnings with autofix suggestions in local dev.

**`hyperflux/enforce-component-factory`**
Every exported React component in application code must be created via `createHyperFluxComponent()` or `createHyperFluxAdapter()`. Violation = build failure.

**`hyperflux/no-literal-content`**
String literals in JSX text content and in configurable JSX props (`label`, `placeholder`, `title`, `message`, `description`, `text`, `heading`, `caption`, and any prop whose value is user-visible) are flagged. Excluded: CSS class names, `aria-*`, `data-*`, `key`, `id`, `name`, `type`, `htmlFor`. Violation = build failure in CI, warning with autofix in dev.

Both rules are shipped in `@hyperflux/eslint` and wired into the Shell's pre-commit hook.

**Architecture checker** (extends `@hyperflux/lint`):
`hf coverage` reports per-file: rule density (useRule calls / configurable values detected), hardcoded literal count, lifecycle address coverage. Not a build gate — a visibility tool. Run in CI and publish as a report.

## 5. `@hyperflux/ui` — The Primitive Library

### 5.1 Scope

Six primitives cover 80% of application UI needs:

| Component | Rule namespace | What comes from rules |
|---|---|---|
| `HFTable` | `{domain}.table.*` | Columns, labels, sort config, empty state message, row actions |
| `HFForm` | `{domain}.form.*` | Field definitions, labels, placeholders, validation rules, submit label |
| `HFModal` | `{domain}.modal.*` | Title, body text, confirm/cancel labels, blocking conditions |
| `HFNav` | `ui.nav.*` | Links, labels, active state logic, visibility per role |
| `HFDetailPanel` | `{domain}.detail.*` | Fields shown, labels, action buttons, section order |
| `HFToolbar` | `{domain}.toolbar.*` | Search placeholder, filter options, button labels |

### 5.2 Source

These components already exist in `apps/admin/hiflux/src/components/`. Phase 1 is extraction and generalization — not invention. The admin runs identically after Phase 1 using `@hyperflux/ui`.

### 5.3 DX Cost — Acknowledged

Writing a rule to define column names is more verbose than writing `columns={["name", "email"]}` inline. This is true. The upfront cost is real.

The rule for when to externalize: if the value might ever change post-deploy, if it appears in more than one component, or if a non-developer needs to control it — it belongs in a rule. If none of those are true, hardcode it and the lint rule will not flag it (it is not a configurable value by definition).

The DX cost drops significantly with two things that ship in Phase 1: type generation (the rules prop is fully typed, autocomplete works) and the admin visual editor (rules are authored in a form UI, not hand-written JSON).

### 5.4 Rule File Organization

No monolithic files. Directory-per-domain, file-per-concern.

```
rules/
  orders/
    table.json        — orders table configuration
    form.json         — order creation/edit form
    policy.json       — who can do what, transition guards
    lifecycle.json    — before/during/after for order operations
  users/
    table.json
    form.json
    policy.json
  shared/
    nav.json
    config.json       — global thresholds, feature flags
  content/
    messages.json     — error messages, empty states, confirmations
```

The engine does not care about file structure. This is a Shell convention. `hf migrate` generates into this structure.

## 6. The State Machine — Complete End-to-End Example

This section shows one full transition: rule definition → lifecycle guard → component reading state → UI update → behavior change without deploy.

**Domain:** Order management. An order moves from `draft` to `submitted`. The transition is only valid if the order has at least one item and payment is confirmed.

**Step 1 — Define the guard rule** (`rules/orders/policy.json`):
```json
{
  "path": "orders.policy.can_submit",
  "kind": "compute",
  "inputs": [
    { "name": "order_status", "type": { "type": "string" } },
    { "name": "item_count",   "type": { "type": "number" } },
    { "name": "payment_confirmed", "type": { "type": "boolean" } }
  ],
  "output": { "type": "boolean" },
  "cases": [
    {
      "when": {
        "op": "AND", "args": [
          { "op": "==", "args": [{ "input": "order_status" }, { "kind": "literal", "value": "draft" }] },
          { "op": ">",  "args": [{ "input": "item_count" },   { "kind": "literal", "value": 0 }] },
          { "input": "payment_confirmed" }
        ]
      },
      "then": { "kind": "literal", "value": true }
    },
    { "then": { "kind": "literal", "value": false } }
  ]
}
```

**Step 2 — Register the lifecycle address** (`lifecycle/manifest.json`):
```json
{
  "orders.submit": {
    "before": {
      "externalizable": true,
      "rules": ["orders.policy.can_submit"]
    },
    "after": {
      "externalizable": true,
      "rules": ["orders.lifecycle.audit_submit"],
      "retry_policy": "none"
    }
  }
}
```

**Step 3 — Component reads state**:
```tsx
const SubmitOrderButton = createHyperFluxComponent<SubmitButtonRules>(
  "orders.submit_button",
  ({ rules, order }) => {
    const canSubmit = useRule("orders.policy.can_submit", {
      order_status: order.status,
      item_count: order.items.length,
      payment_confirmed: order.paymentConfirmed,
    });
    return (
      <button disabled={!canSubmit}>
        {rules.label}
      </button>
    );
  }
);
```

**Step 4 — Server enforces the transition** (API route):
```typescript
const engine = createLifecycleEngine();
try {
  await engine.run("orders.submit", "before", {
    order_status: order.status,
    item_count: order.items.length,
    payment_confirmed: order.paymentConfirmed,
  });
} catch (err) {
  if (err instanceof LifecycleBlockedError) {
    return NextResponse.json({ error: err.reason }, { status: 422 });
  }
  throw err;
}
// proceed with submit logic
await engine.run("orders.submit", "after", { order_id: order.id });
```

**Step 5 — Behavior change without deploy**

A business rule changes: orders over $10,000 now also require manager approval before submission. A product manager logs into the admin, edits `orders.policy.can_submit`, adds a new input `requires_approval` and a new condition. Saves. The override is written to the KV store. On the next request, every component reading this rule and every lifecycle guard evaluating it sees the new behavior. No PR. No build. No deploy.

This is the complete loop.

## 7. Runtime Error Model

When a component reads a rule path that does not exist, the resolver throws `RuleNotFoundError`. The factory catches this and applies the following behavior — non-configurable:

- Returns the component's declared fallback value
- In development: fires a console warning with the exact missing path and a stack trace pointing to the component
- In production: logs silently, component renders with fallback
- Never throws to the React error boundary
- Never crashes the UI

A missing rule path is a developer error, not a user error. The user sees a fallback. The developer sees a warning. The application continues running.

The factory signature requires a fallback value per rule namespace. Components that do not declare fallbacks for required rules fail TypeScript compilation.

## 8. Type Generation

`hf generate-types` is a Phase 1 deliverable. It ships alongside the factory. They are the same feature.

**How it works:**

1. Scans the codebase for all `createHyperFluxComponent<T>(namespace, ...)` calls
2. Reads the rule files for each namespace
3. Reads each rule's `TypeSpec` output definition
4. Emits a `.d.ts` file per domain

**Output example** (generated, not hand-written):
```typescript
// generated/rules/orders.d.ts
export interface OrdersSubmitButtonRules {
  label: string;
  disabled_reason: string;
}
export interface OrdersTableRules {
  columns: string[];
  page_size: number;
  empty_state_message: string;
}
```

**Integration:** Runs as a pre-build step (`hf generate-types --watch` in dev). The factory's type parameter is validated against the generated interface at compile time. The `rules` prop inside any component is never typed as `any`.

Without type generation, the factory is worse DX than plain React. With it, it is better — full autocomplete, compile-time validation of rule paths, no runtime surprises.

## 9. Performance Model

**Rule resolution cost:**

The resolver caches by `canonicalJSON(path + inputs)`. A component reading `useRule("orders.policy.can_submit", { order_status: "draft", item_count: 3, payment_confirmed: true })` with stable inputs hits the cache on every re-render. Cost: one Map lookup per rule per render. Negligible.

Cold resolution (first render or changed inputs) runs the full resolution tree. Cost: O(depth of rule dependency graph). For rules with no dependencies (most content rules), this is O(1). For complex computed rules, O(d) where d is dependency depth.

**Factory-level memoization (mandatory, built in Phase 1):**

- `React.memo` on the component wrapper — prevents re-render when parent re-renders with same props
- `useMemo` on the inputs object — prevents new object reference on each render from triggering resolver cache miss
- Selective rule subscription — a component re-renders only when its declared rule namespace changes, not when any rule changes

**Benchmark target:** 50 HyperFlux components on screen, each reading 3 rules, stable inputs. Measured re-render cost must be ≤ equivalent plain React component tree. This is measured in Phase 1 before Phase 2 begins. If it fails, Phase 2 does not start.

## 11. hf migrate — Corrected Scope

**Moved to Phase 2. Exit criteria changed.**

**What it does:** Scans an existing React/TypeScript application. Detects string literals, numeric thresholds, boolean flags, and conditional UI logic that are candidates for externalization. Generates draft rule files. Produces a PR with `status: "draft"` on every generated rule.

**What it does not do:** Apply rules automatically without review. Guarantee zero manual edits. Handle arbitrary code patterns the AI cannot classify.

**Exit criteria (replacing "no manual edits"):**  
Run `hf migrate` against `benchmark/vanilla/`. Measure: what percentage of the 20 benchmark change-points are correctly identified and extracted as draft rules? Target ≥ 80% correct identification. The remaining ≤ 20% are flagged with explicit reasons. A developer can review and promote all generated rules in ≤ 30 minutes.

**Trust model:** Generated rules are proposals, not facts. The admin's promote-to-source flow (already built) is the human gate. The AI proposes. The developer promotes. Nothing goes active without explicit human approval.

**Confidence scoring:** Each generated rule carries a confidence score. High confidence (isolated string literal, clear label semantics) → auto-drafted. Low confidence (complex expression, ambiguous intent) → flagged for human decision with the AI's reasoning shown. The developer only reviews the ambiguous cases.

## 12. Escape Hatches — The Pluggable Action Pattern

An escape hatch that is easy to use will become the default. An escape hatch that is expensive and visible will be used only when genuinely necessary.

**The pattern:** Rules reference named functions by ID. The function is registered in code. The WHEN the function runs is in the rule. The HOW is in code.

This already exists in HyperFlux's expression language as `FnExpression`. A rule case can call a registered function by name. The orchestration (when to call it, what inputs to pass, what to do with the result) remains in the rule. The implementation lives in code where it is debuggable and testable.

```json
{
  "path": "orders.policy.validate_address",
  "cases": [{
    "then": { "kind": "fn", "fn": "validateShippingAddress", "args": [{ "input": "address" }] }
  }]
}
```

```typescript
// Registered in code — implementation only, no orchestration
functionRegistry.register("validateShippingAddress", async (address) => {
  return externalValidationService.check(address);
});
```

**What is not allowed:** A `customValidator` prop on a component. A `customRenderer` prop. Any prop that accepts a function defined at the call site. These put decisions back in code. The lint rule flags any component prop typed as a function that is not a registered action reference.

## 13. Delivery Plan

- Implement `RuleOverrideStore` adapter interface

- Pre-commit hook wiring for `@hyperflux/eslint`

- Run the full 20-change benchmark against Shell

- **Exit criteria:** Actual measurement published. If below 80%, the gap is understood and a Phase 5 is defined.

## 14. Closed Questions

All open questions from v0.1 are closed:

1. **Factory vs HOC vs decorator** → Factory. Closed.
2. **Escape hatch for domain-specific logic** → Pluggable Action via `FnExpression`. Closed.
3. **ESLint severity** → Warning + autofix in local dev. Hard error in CI. Closed.
4. **Third-party components** → Value-source contract enforced at call site. `createHyperFluxAdapter()` for repeated patterns. Closed.
5. **Versioning between `@hyperflux/ui` and `@hyperflux/core`** → Semantic versioning. `@hyperflux/ui` can release independently of Core. Breaking changes in Core require a major version bump. `@hyperflux/ui` declares a peer dependency range on Core. Closed.

## 15. What This Does Not Do

- Does not change `@hyperflux/core`
- Does not generate backend code
- Does not replace a design system — `@hyperflux/ui` is structural; it wraps shadcn, Tailwind, or any design system
- Does not enforce backend state machines — scope is the UI and decision layer
- Does not guarantee zero bugs in migrated applications — `hf migrate` proposes, humans decide
